# 本地短时活动规划与执行 Agent（规划器 / 工具调用设计文档）

# MVP版本完整PRD 

## 一、业务背景（精简）

**核心痛点**：用户在 3\-4 小时短时出行场景中，需要在多 App 之间手动串联位置、排队、天气、门票信息，决策成本极高，大量长尾出行需求在决策阶段直接流失。

**为什么用 AI**：用户意图高度模糊（\&\#34;带孩子别太折腾\&\#34;），传统规则引擎无法处理；现实动态不确定（排队暴增、突发暴雨），需要 Planning → Reflection → Replanning 闭环。

**核心分工**：LLM 做意图解析与动态损益评分，Python/Go 做确定性路线计算与 API 调用。

**MVP 边界**：不做真实支付扣款；不接入真实高德/美团线上 API，全量使用本期 Mock 数据池；不做多端实时协同。

### 3\.3 阶段二：CoT 动态规划 \+ 草稿态画布共建

#### CoT 动态思考过程展示（新增）

规划阶段 Agent 的推理过程**实时流式展示**，而非让用户面对空白等待。交互形态参考 Claude Extended Thinking 模式：

**展示形态**：

```Plain Text
🤔 正在为您规划...

▼ 思考过程（可展开）
  ├─ 识别到：家庭场景，5岁孩子，怕晒，2点出发
  ├─ 约束提取：室内优先，3公里内，儿童友好必须
  ├─ 排除：户外公园（曝晒风险）、长排队场所（孩子易哭闹）
  ├─ 候选评分中：乐高体验馆 92分 / 绘本探索馆 87分 / 儿童科技馆 81分
  └─ 餐厅筛选：排除排队>20min，优先 child_seat 标签...

```

**规则**：

- 思考过程默认**折叠**，显示\&\#34;查看规划思路 ▼\&\#34;入口

- 用户点击可展开查看完整 CoT 推理，包括评分明细和排除理由

- 流式输出，逐步追加，不等全部推理完成再展示

- CoT 内容必须真实反映 LLM 的 `thought` 字段内容，不得伪造摘要

**Prompt 约束**：所有 LLM 节点的 `thought` 字段内容必须：

1. 先识别场景与特殊人员

2. 列出关键约束与排除项

3. 给出评分排序与选择理由

4. 最终输出前完成行程合理性自检（详见第五章）

## 五、行程合理性校验（独立章节）

### 5\.1 为什么需要独立的合理性校验

LLM 具备强大的语义理解能力，但在处理\&\#34;看起来可行，实则体验极差\&\#34;的方案时容易失效。典型问题是：**所有单点约束都满足，但整体体验严重翻车**。

**反面案例（高德真实 BadCase）**：

> 用户问\&\#34;周末想游山玩水\&\#34;，被推荐京郊青龙峡。用户说\&\#34;没有车\&\#34;（隐含：太远了），系统没有确认意图、没有重新推荐，而是直接优化原方案，让用户坐公交。用户说\&\#34;时间不够\&\#34;，系统仍未确认，继续查询公交班次和景点开门时间，最终给出\&\#34;可行\&\#34;结论：公交往返 6 小时，游玩时间仅 1 小时。营业时间✓、公交路线✓、物理上\&\#34;来得及\&\#34;✓，但通勤/游玩比例 = 6:1，这是一次彻底的体验翻车。
> 
> 

**核心原则**：Agent 必须在输出方案前完成合理性自检，**拒绝输出看起来合法但体验极差的方案**。同时，当用户补充信息（如\&\#34;没有车\&\#34;）时，必须先**确认意图**，而不是直接执行推断出的子任务。

### 5\.2 合理性校验规则集

以下规则在 LLM 的 `thought` 字段中强制执行，任何一条不通过则方案不得输出，必须重新规划：

**规则一：通勤/游玩时间比例**

- 单程交通耗时不得超过总出行时长的 30%

- 往返交通耗时合计不得超过总出行时长的 40%

- 违反示例：总行程 4 小时，往返通勤 2\.5 小时 → 不合格，必须换更近的目的地

**规则二：核心活动时间保障**

- 每个主要活动节点的有效游玩时间不得少于 45 分钟（不含交通）

- 含儿童场景：儿童活动时间不得少于 1 小时，避免\&\#34;赶场\&\#34;

- 违反示例：游玩时间仅 1 小时但通勤 6 小时 → 不合格

**规则三：交通方式与距离匹配**

- 无车用户（步行/公交）：单程距离超过 15 公里的目的地禁止推荐

- 步行出行：单程超过 2 公里禁止推荐步行方式，必须建议打车/骑行

- 含儿童：单程步行超过 800 米需标注风险

**规则四：意图确认优先于方案优化**

- 当用户的补充信息（\&\#34;没有车\&\#34;、\&\#34;时间不够\&\#34;、\&\#34;太远了\&\#34;）与当前方案存在根本冲突时，**必须先确认用户意图**（是否需要重新选目的地），而不是直接在原方案上优化

- 禁止行为：用户说\&\#34;太远了\&\#34; → 系统查公交路线而不重新推荐

**规则五：节奏合理性**

- 全天行程节点数：建议 2\-4 个主活动 \+ 1 顿饭，最多不超过 6 个节点

- 相邻节点间必须有合理交通缓冲时间（不能规划 16:00 乐高结束，16:01 到达餐厅）

- 含儿童：行程末节点需在晚 20:00 前结束，避免晚归

**规则六：季节与天气适配**

- 雨天 / 极端高温（\&gt;37°C）/ 极端低温（\&lt;0°C）：禁止推荐长时间户外活动

- 当前天气 = heavy\_rain 时，户外节点自动触发风险警告

### 5\.3 合理性自检在 CoT 中的呈现

LLM 在输出方案前，`thought` 字段必须包含合理性自检段落：

```Plain Text
合理性自检：
- 通勤比例：往返约 40min / 总时长 240min = 17% ✓
- 核心活动时间：乐高 90min ✓ / 餐厅 60min ✓
- 步行距离：乐高→餐厅 约 800m，建议打车提示 ⚠️
- 节点数量：3个节点 ✓
- 儿童结束时间：预计 19:30 结束 ✓
结论：方案合格，可输出
```

## 六、系统 Workflow 工作流编排

```Plain Text
[用户输入 / 追加操作]
         │
         ▼
(1. LLM 意图与操作解析)  ──► [CoT 流式展示给用户]
         │
         ▼
(2. 确定性并行工具链)  ──► /api/location + /api/weather + /api/activities/search
         │
         ▼
(3. LLM 损益评分 + 行程合理性自检)  ──► [CoT 流式展示 评分过程]
         │
         ├── 合理性不通过 ──► 重新规划，禁止输出
         │
         ▼
(4. 草稿拼装渲染)  ◄──── 用户编辑循环
         │
         │ 用户点击 [一键履约]
         ▼
(5. 资源策略系统)  ──► 根据资源类型决定软锁时机
         │
         ▼
(6. 并行履约引擎)  ──► 工具调用前中后状态检查
         │
         ▼
(7. 看板生成 + 伴随监控)  ──► [突发异常] ──► (8. 局部重规划路由)
                                                        │
                                                        ▼
                                              [知情权弹窗 → 用户确认]
```

### 节点 1：LLM 意图与操作解析器

输入：用户自然语言 / 草稿阶段追加操作

识别两类信息：

1. **实体与约束**：scene / companions / time / location / preferences

2. **操作语义**：OPERATION = INSERT / DELETE / REPLACE / PIN / UNPIN / DELAY / SKIP

**意图冲突检测**：当用户补充信息与当前方案存在根本冲突时（如\&\#34;没有车\&\#34;对应远郊景区），解析器输出 `intent\_conflict: true`，触发意图再确认而非方案优化。

输出示例：

```JSON
{
  "scene": "family",
  "companions": ["adult_female", "child_age_5"],
  "trip_rigidity": "high",
  "start_time": "14:00",
  "preferences": ["indoor", "avoid_sun"],
  "intent_conflict": false,
  "operation": {
    "action_type": "INSERT_NODE",
    "target_category": "EV_CHARGING",
    "insert_after_node_id": "node_001_lego",
    "constraints": { "connector_type": "FAST", "distance_limit_km": 2 }
  }
}
```

### 节点 2：确定性并行工具链

并行调用：`/api/location` / `/api/weather` / `/api/activities/search`

输出：纯事实数据池（Raw Fact Pool），不含任何 LLM 推断

### 节点 3：LLM 损益评分 \+ 合理性自检

输入：Raw Fact Pool \+ 用户约束 \+ 当前模式（`mode: light \| full`）

必须在 `thought` 字段中完成：① 模式确认 → ② 特殊人员约束 → ③ POI 评分排序 → ④ 行程合理性自检

输出：候选 POI 排行榜 \+ `planner\_reason` \+ `feasibility\_check`

### 节点 4：确定性路由

判断条件：存在 `soft\_locked == true` 或 `user\_pinned == true` 的节点

- YES → 局部重规划器（仅重构 pending 节点，绝对保护锁定节点）

- NO → 完整行程生成器

### 8\.2 资源策略系统（软锁差异化）

不同类型资源的锁定时机根本不同，不能使用同一套策略：

**景点 / 演出类（稀缺资格型）**

- 特征：门票稀缺、提前放票、取消有损失（退款难 / 资格释放）

- 软锁触发时机：**计划生成时即触发软锁**

- 软锁内容：提示用户\&\#34;此景点门票较难抢，建议立即预约以锁定资格\&\#34;

- 取消时触发资源损失警告（见第九章）

**餐厅类（动态时间型）**

- 特征：不稀缺、最优预约时机是出发前 1\-1\.5 小时（太早不如不约）

- 软锁触发时机：**动态计算**，在出发前约 90 分钟、排队预测超过阈值时触发

- 软锁内容：\&\#34;距离用餐约 90 分钟，现在是最佳取号时机，是否帮您取号？\&\#34;

- 不应在计划生成时就软锁餐厅，否则占用排队资格又浪费

**充电桩 / 停车类（实时可用型）**

- 特征：完全实时，提前锁定无意义

- 软锁触发时机：**出发前 15 分钟**才触发

- 不需要早锁

**软锁触发公式**：`软锁优先级分 = 稀缺系数 × 替代难度 × 取消损失系数`

## 九、资源损失警告 \+ 用户主动上报冲突

### 9\.1 资源损失警告（Resource Loss Warning）

当用户尝试取消或替换一个**已有成本的节点**（已预约、已取号、已付款、余票稀缺）时，必须在执行前弹出损失提醒：

**弹窗示例**：

> 取消当前计划后：
> 
> - 故宫预约资格将立即释放
> 
> - 当前时段余票已售罄，再次预约成功率极低
> 
> - 后续可约日期仅剩 2 天
> 
> 是否确认取消？ \[确认取消\]　\[保留，我再想想\]
> 
> 

**触发条件**：

- 节点状态为 soft\_locked 或 completed\_lock

- 节点存在已付款订单

- 节点 resource\_type = \&\#34;attraction\&\#34; 且稀缺系数 \&gt; 0\.7

- 排队资格已占用（已取号）

**禁止行为**：在未展示损失警告的情况下直接执行取消或替换。

### 9\.2 用户主动上报冲突（半结构化流）

用户点击卡片 \[出问题了\] 后，进入以下交互流程：

**第一步：多选原因（系统提供选项，减少输入负担）**

```Plain Text
遇到什么问题了？（可多选）
☐ 排队太久了
☐ 活动时间超了
☐ 下雨 / 天气不好
☐ 孩子/朋友不想去了
☐ 地方找不到 / 路线有问题
☐ 其他（自己说）
```

**第二步**：用户选择\&\#34;其他\&\#34;时，弹出自然语言输入框，用户用文字描述问题。

**第三步**：系统识别冲突类型，给出 2\-4 个恢复方案（按推荐顺序排列），用户选择一个执行：

```Plain Text
根据您的情况，为您准备了以下方案：

① 换【同商圈亲子主题餐厅】，当前排队 5 分钟，步行 3 分钟可达  ⭐推荐

[选择方案①]　[选择方案②]　[选择方案③]　[都不合适，我自己调]
```

**禁止行为**：选项不超过 4 个，不给用户\&\#34;无限可能\&\#34;，减少选择负担。

### 12\.2 十类失败场景处理

**场景 1：AI 生成失败**（大模型无返回 / JSON 解析失败）

- 重试 1 次 → 仍失败 → 使用规则模板生成基础行程（附近活动 → 餐厅 → 回家）

**场景 2：POI 匹配失败**（搜索无结果）

- 扩大搜索范围 → 换关键词（亲子乐园 → 儿童乐园 / 商场儿童区）→ 推荐同类型备选

**场景 3：餐厅不可用**（满座 / 排队过长 / 暂停营业）

- 同类替换 → 同商圈替换 → 降低复杂度（改为轻食快餐）

**场景 4：预约/下单失败**（接口失败 / 余票不足）

- 重试 → 换时段 → 换同类 POI → 提示用户手动处理

- **严禁伪装成功**

**场景 5：执行超时**（当前活动超时，后续来不及）

- 压缩后续活动 → 跳过非关键节点 → 延后餐饮/预约 → 重规划剩余行程

**场景 6：用户临时变更**（孩子累了 / 朋友想换地方）

- 保留 completed\_lock 节点 → 保护 user\_pinned 节点 → 只重规划 pending 节点

**场景 7：软锁节点被取消**（用户想取消有成本节点）

- 弹出资源损失警告 → 用户确认后才执行 → 取消后自动重规划后续

**场景 8：定位/地图失败**

- 使用用户手动输入位置 → 商圈级推荐 → 不计算精确路线，给附近方案

**场景 9：全局计划崩溃**（下雨 \+ 多节点不可用 \+ 晚出发）

- 触发全局恢复 → 重规划剩余时间 → 生成低风险室内兜底方案

**场景 10：Mock 数据池备选穷尽**

- 展示\&\#34;附近符合条件的备选已用完\&\#34; → 提供 \[扩大到 5 公里\] / \[换个类型\] 快捷选项

- 不允许 LLM 编造不存在的 POI

# **3\. Agent 核心设计目标**



## **3\.1 低决策成本**

减少用户：

- 搜索成本

- 多 App 切换

- 多人协商成本

- 反复调整成本



系统优先生成：

> “稳定、省心、低风险”的方案。
> 
> 

而不是最复杂或最热门方案。

## **3\.2 现实可执行性优先**

规划器不仅生成行程（itinerary），还必须完成现实对齐：

- 营业时间校验

- 排队情况校验

- 天气校验

- 距离校验

- 库存 / 预约校验

避免：

> “理论最优，但现实翻车”。
> 
> 



## **3\.3 可动态调整**

系统支持：

- 用户中途改主意

- 天气变化

- 排队过长

- 活动不可执行

并通过动态重规划：



> 只修改未来节点，不推翻整个计划。
> 
> 

# **4\. 系统架构**



系统采用：



```Plain Text
Tool Layer（事实层）
↓
Planner Layer（决策层）
↓
Execution Layer（履约层）
```



## **4\.1 Tool Layer（事实层）**



工具层只负责：



- 查询

- 获取

- 返回事实数据

不进行主观判断。



例如：



- 排队时间

- 距离

- 营业时间

- 天气

- 库存

- 评分

工具层不返回：



- “高风险”

- “适合家庭”

- “推荐”

- “不适合”

这些属于 Planner 的推理结果。



**\-\-\-**



## **4\.2 Planner Layer（决策层）**



Planner 负责：



- 场景理解

- 约束推理

- 风险计算

- 候选排序

- fallback

- replanning

例如：



```Plain Text
家庭场景
+
孩子同行
+
queue_time = 90min
↓
不适合作为首选餐厅
```



**\-\-\-**



## **4\.3 Execution Layer（履约层）**



负责：



- 预约

- 排队取号

- 模拟支付

- 履约状态同步

- 风险提醒

**\-\-\-**



# **5\. 规划器工作流**



## **5\.1 整体流程**



```Plain Text
[1] 用户输入
↓
[2] 意图解析
↓
[3] 上下文补全
↓
[4] 候选搜索
↓
[5] 现实对齐
↓
[6] 候选评分
↓
[7] 行程生成
↓
[8] 用户微调
↓
[9] 执行履约
↓
[10] 履约状态流
↓
[11] 动态重规划 / fallback
```



**\-\-\-**



## **5\.2 输入理解**



系统解析：



- 时间范围

- 场景类型

- 用户意图

- 出行约束

示例：



```Plain Text
今天下午想带老婆孩子出去玩
```



解析：



```JSON
{
  "scene": "family",
  "time": "afternoon",
  "distance_constraint": "nearby"
}
```



**\-\-\-**



## **5\.3 上下文补全**



部分信息默认推断。



部分信息必须追问。



### **必须追问**



|信息|原因|
|---|---|
|出发时间|影响营业时间与排队|
|是否有小孩 / 老人|影响风险策略|



### **自动获取**



|信息|来源|
|---|---|
|用户当前位置|定位工具|
|天气|天气工具|



### **默认约束**



家庭场景：



- 少走路

- 少排队

- 儿童友好

- 节奏轻松

朋友场景：



- 社交

- 拍照

- 热门

- 性价比

系统先生成初版方案，再允许用户补充信息与微调。



**\-\-\-**



# **6\. 工具调用设计**



系统采用：



> 单规划 Agent \+ 多工具
> 
> 



避免多 Agent 编排复杂度。



**\-\-\-**



# **6\.1 工具列表**



## **1）用户上下文解析工具**

- 场景

- 用户意图

- 时间

- 约束

- 名称

- 热度

不返回：



- “适合家庭”

```Plain Text
+
孩子同行
+
queue_time = 90
↓
不适合作为家庭首选
```



**\-\-\-**



## **4）天气查询工具**



返回：



- 天气

- 温度

- 是否适合户外

**\-\-\-**



## **5）路线查询工具**



返回：



- 距离

- 通勤时间

- 步行成本

**\-\-\-**



## **6）候选评分工具**



独立评分模块。



负责：



- 候选排序

- 风险计算

- fallback 决策

- 替换节点排序

**\-\-\-**



## **7）行程生成工具**



生成完整行程。



固定结构：



```Plain Text

```



**\-\-\-**



## **8）统一履约工具**



支持：



- 预约

- 排队取号

- 模拟支付

**\-\-\-**



## **9）动态重规划工具**



用于：



- fallback

- 节点替换

- 未来节点重规划

**\-\-\-**



# **7\. 工具调用策略**



系统采用：



> 部分并行工具调用。
> 
> 



## **并行调用**



同时查询：



- 天气

- 活动

- 餐厅

- 用户位置

## **后置调用**



路线与时间规划后置：



```Plain Text
候选生成
↓
路线计算
↓
综合评分
```



这样可以减少等待时间，并增强 Agent 的“实时工作感”。



**\-\-\-**



# **8\. 候选评分策略**



系统固定 6 个评分维度：



|维度|说明|
|---|---|
|距离便利性|是否离用户近|
|排队 / 等待成本|是否容易踩坑|
|用户偏好匹配|是否符合场景|
|节奏舒适度|是否轻松|
|现实可执行性|是否能顺利完成|
|热门 / 价格|社交与性价比|



**\-\-\-**



## **动态权重**



### **家庭场景提高：**



- 距离

- 排队

- 舒适度

### **朋友场景提高：**



- 热门程度

- 拍照氛围

- 性价比

**\-\-\-**



# **9\. 动态重规划机制**



## **9\.1 核心原则**



```Plain Text
保护已完成
保护已预约
只修改未来节点
```



系统避免：



> “用户改一个点，整个行程崩掉”。
> 
> 



**\-\-\-**



## **9\.2 软锁机制**



以下资源进入软锁状态：



- 已预约餐厅

- 已支付门票

- 已排队取号

- 已完成事项

修改前必须：



- 风险提醒

- 用户确认

路线不进入软锁，可动态刷新。



**\-\-\-**



# **10\. fallback 机制**



## **10\.1 排队过长**



家庭场景主演示：



```Plain Text
餐厅排队90分钟
↓
系统检测高等待成本
↓
推荐同商圈备选
↓
用户确认
↓
重新预约
```



**\-\-\-**



## **10\.2 天气变化**



朋友场景主演示：



```Plain Text
天气变化
↓
户外活动风险提醒
↓
推荐室内活动
↓
用户确认
↓
替换未来节点
```



**\-\-\-**



## **10\.3 用户改主意**



例如：



```Plain Text
孩子累了
不想吃火锅了
太远了
```



系统：



```Plain Text
锁定已完成节点
↓
分析受影响范围
↓
仅修改未来节点
↓
重新生成后续行程
```



**\-\-\-**



# **11\. 履约状态流**



系统在规划与履约过程中展示状态流：




- 降低用户焦虑

# 前端 UI 骨架与交互状态设计

## 整体布局和交互设计（对话\+动态卡片流）

为了解决传统 AI 纯对话模式下“用户反复确认效率低、在聊天和商品页来回切换”的问题，本产品在前端 UI 上完全对齐美团原生移动端的用户习惯，AI 在这里主要充当流程组织者的角色，用户通过语音或文字输入需求后，页面内直接生成、渲染对应的结果，实现“一句话提出需求 $\rightarrow$ 自动生成方案 $\rightarrow$ 手动局部微调 $\rightarrow$ 直接下单核销”的极简流程。

## 1\.1聊天流组件骨架设计

聊天流位于整个手机屏幕的最上方，是用户输入需求和系统锁定核心目标的区域。对应原型图中的 **① 用户需求区**。



### 1\.1\.1 页面盒子与排版布局

- **外层大盒子**：在首屏最顶部设计一个 DIV容器，固定宽度 width：100%，四周留出标准的内边距，背景用纯白色，带一点点圆角，边框用很浅的灰色线扎边，显得干净专业。

- **左侧文本区**：盒子里面的左边放一个文本标签，用来展示用户最初说的那句模糊需求。字体用稍微粗一点的黑色字，保证显眼。  

- **右上角状态图标**：盒子的右上角放一个“小蓝锁”的图标，文案标注 （已锁定🔒），明确告诉用户当前行程是基于这个需求生成的。  

- **右侧操作按钮**：在小蓝锁下面放一个带边框的按钮，里面带一个小铅笔图标，文字写 **【编辑需求】**。  



### 1\.1\.2 交互逻辑与 Demo 假字占位

- **静态展示状态**：前端页面刚加载出来时，左侧文本区展示下面这段 Mock 需求文字：

> - “今天下午带5岁孩子出去玩，不要太远，老婆在减肥需要清淡饮食，4\-6小时”
> 
> 

- **点击编辑交互**：用户用手指去点【编辑需求】按钮时，这个大盒子立刻切换成输入框模式（把文本变成可输入的 textarea），小蓝锁变成开锁状态。用户改完字点击“保存”后，整个盒子重新锁定，同时往后台发送请求，把下面的行程卡片流全部刷新（全局重规划）。  



## 1\.2 状态流组件骨架设计

状态流由两部分组成，一部分是动态加载时的‘异步履约状态流，’另一部分是执行时的“计划健康提示层”，对应原型图中的 **② 计划健康层**。



### 1\.2\.1 页面盒子与排版布局

- **外层大盒子**：紧跟在聊天流盒子的正下方。平时如果行程没出问题，这个盒子高度设为 0，直接隐藏。

- **背景颜色分级**：当有通知时，盒子动态弹出来。颜色严格做成两种：

    - *黄色（风险卡）*：背景用淡淡的橙黄色，字用深橘色，代表有潜在风险，但行程还能走。  

    - *红色（冲突卡）*：背景用淡淡的粉红色，字用大红色，代表严重冲突，原计划肯定泡汤了。  

- **内部排版**：盒子左边放一个警告感叹号的小图标，中间放大白话的风险摘要文字，右边放一个向右的小箭头\&gt;，提示用户可以点击看详情。  



### 1\.2\.2 交互逻辑与 Demo 假字占位

- **异步加载时的流水账提示（后台算数据时显示）**：

    - 在 Demo 演示 AI 正在后台快速查美团 API 的那段时间里，状态栏会一行一行淡出下面这些提示字，让用户知道 AI 在干嘛，心理上有安全感：  

    - 正在帮您安排下午的行程，预计还需20秒\.\.\.

    - ✓ 成功认领需求：已开启\[5岁带娃\]特殊照顾与\[突发暴雨\]避雨策略。  

    - ✓ 自动拦截风险：已一键屏蔽全城所有室外公园、露天广场、高强度运动场所\(\)。  

    - ⏳ 正在查美团数据：已过滤掉受下雨影响的POI（地图上具体的地点），正在核对万象城商圈室内场馆的门票和排队情况\.\.\.

- **运行态时的突发风险提示（对应原型图效果）**：

- 行程出发后，如果后台数据变了，该层立刻变黄并弹出来，字符占位如下：

    - ⚠️ 后续 1 项活动存在时间风险：晚饭可能会迟到，建议提前出发或调整预约时间。  

用户拿手指戳这行字，就能直接弹出具体的解决办法。



## 1\.3 Itinerary 行程卡片流组件骨架设计

行程卡片流是整个手机屏幕上占地方最大、内容最硬核的核心主体，由一个运行状态大盒子和一长串纵向时间轴卡片串联组成，对应原型图中的 **③ 运行状态层** 和 **④ 任务流程卡片**。



### 1\.3\.1 运行状态层盒子（当前进行中 / 上方常驻区）

- 排版骨架：位于状态流正下方。盒子用细绿线勾边，左上角打上绿色的 当前进行中 标签，右上角打上 进行中 状态字。  

- 内部上下分层：

    - 上半截（当前任务）：大字写 亲子乐园（欢乐海洋馆），右边配一张海洋馆的小缩略图，下面写上一行小字 预计结束 15:40。 

    - 下半截（下一计划预告）：背景用浅灰色稍微和上面隔开。左边画一个小叉子勺子的餐饮图标 🍴，写上 下一计划：16:10 海底捞火锅（万象城店）。右边写上辅助的大白话提醒：建议最晚 15:55 出发 \| 距 4\.2km，预计 20 分钟。  

- 底部三个快捷 GUI 按钮（核心交互点）：

    在盒子最底部，并排整齐地摆放三个大按钮，用户手指极易点击：  

    - 【导航】：绿色字。点一下直接在页面内划出路线，算出到海底捞还有多少米、开车要多久。  

    - 【✓ 完成打卡】：绿色背景白字。用户在现场玩完了点一下，这个进行中盒子就会刷新，把海底捞火锅提到上面变成新的“进行中”，原来的海洋馆卡片自动归档变灰。  

    - 【⚠️ 异常/求助】：橙色字。用户要是现场突发意外（比如孩子闹脾气不肯走，或者外面雨越下越大），点这个按钮，界面会立刻弹出一个多选框（选项包括：排队太久、突发下雨、时间超了、临时改主意等），让用户选完后直接触发后台重规划。  



### 1\.3\.2 任务流程卡片（纵向时间轴流）

- **纵向虚线骨架**：在运行状态层下方，要拉出一条由上到下贯穿全屏的灰色虚线（时间轴连线），线上每一个节点对应一个圆圈时间点。  

- **卡片盒子的四种 Mock 状态视觉**：

    **进行中卡片（对应时间 14:00\~15:40）**：边框用蓝色或绿色高亮\(\)。  

    - *字符与标签占位*：大字写 **亲子乐园（欢乐海洋馆）**。下面并排三个小标签 适合亲子、室内、雨天友好。右上角打上蓝色小字标签 门票已购。  

    **风险/预约卡片（对应时间 16:10\~17:30）**：边框用醒目的淡橘黄色。  

    - *字符与标签占位*：大字写 **海底捞火锅（万象城店）**。下方写亮点信息 清淡锅底 · 已提前预约。卡片底部最重要的地方展示实时监控字样：当前排队：12桌（约20分钟）。右上角打上浅色状态标签 已预约。

    **未开始卡片（对应时间 18:00 以后的所有卡片）**：标准的普通白色背景盒子，代表还没发生的行程。  

    - *字符占位一*：18:00\~18:40 \| 江边散步（滨江公园） \| 休闲散步 看日落。  

    - *字符占位二*：19:00\~20:00 \| 轻食晚餐（绿茶餐厅） \| 清淡健康 · 适合减脂。  

    - *折叠交互*：如果卡片太多，底部留一个文字链接 **【查看后续 2 项活动 】**，点一下能把后面的轻食晚餐卡片展开或者收起。  

    **已完成卡片（置灰归档状态）**：

    - 只要用户在上面点了“完成打卡”，或者是时间已经过去的地方，卡片整体背景变灰，字变浅，变成不可修改的只读状态，方便用户随时往回看，心里踏实。
