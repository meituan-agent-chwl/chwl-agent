## 用例 1：家庭场景 — 完整链路测试

**目的**：验证从输入到履约完成的全流程

| 步骤 | 用户输入 | 预期系统行为 | 验收标准 |
|------|---------|-------------|---------|
| 1.1 | 自动启动（无需输入） | 规划过程中显示 `[.] 正在获取当前位置...` 等状态流 | ✅ 状态流逐步展示 |
| 1.2 | 等待 3 秒 | 管家显示行程方案卡片：3 个节点（活动 + 餐厅 + 轻活动），含时间、标签 | ✅ 显示 3 个节点 |
| 1.3 | `确认` | 管家回复"开始预约和取号"，履约进度逐条显示 | ✅ 所有节点最终变成 [OK] |
| 1.4 | `看看方案` | 管家显示行程卡片 + "已完成"状态 | ✅ states 为 completed |

**通过条件**：1.2 看到 3 个节点卡片，1.3 履约全部成功

## 用例 2：家庭场景 — 异常重规划

**目的**：验证 System 能检测异常并触发重规划

| 步骤 | 用户输入 | 预期系统行为 |
|------|---------|-------------|
| 2.1 | 等待规划完成 | 显示行程方案 |
| 2.2 | `孩子累了` | 管家回复"收到反馈，调整方案"，显示调整方案提示 |
| 2.3 | `确认` | 原方案已调整，重新安排 |

**通过条件**：输入"孩子累了"后，System 响应且不崩溃

## 用例 5：队友 API 模式

**目的**：验证切换到队友 Mock API 后能正常工作

```cmd
# 终端 1：启动队友 API
cd C:\Users\hgghhy\meituan-agent\mock_api
python -B app.py

# 终端 2：启动 CLI
cd C:\Users\hgghhy\meituan-agent
venv\Scripts\activate
python scripts\cli_demo.py --teammate
```

**检查点**：
- ⚠️ 队友 API 需要在另一个终端先启动
- 规划方案中的活动/餐厅数据来自队友的 API
- 履约、取消等逻辑应与纯内存模式一致
