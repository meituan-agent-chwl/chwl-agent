> 修订说明（v3）：在 v2 基础上新增/修订以下内容：① 需求清单改为 P0/P1/P2 三级分级，社交分享升入 P0；② 新增 CoT 动态思考过程展示机制；③ 行程合理性校验独立成章并补充通勤/游玩比例等规则及 BadCase；④ 新增资源损失警告、资源策略系统（软锁差异化）、用户主动上报冲突流程、安全机制、Fallback 分层体系；⑤ 大幅扩充 Good/Bad Case 库；⑥ 评测体系增加 LLM Judge 与冲突恢复维度。

## 一、业务背景（精简）

**核心痛点**：用户在 3-4 小时短时出行场景中，需要在多 App 之间手动串联位置、排队、天气、门票信息，决策成本极高，大量长尾出行需求在决策阶段直接流失。

**为什么用 AI**：用户意图高度模糊（"带孩子别太折腾"），传统规则引擎无法处理；现实动态不确定（排队暴增、突发暴雨），需要 Planning → Reflection → Replanning 闭环。

**核心分工**：LLM 做意图解析与动态损益评分，Python/Go 做确定性路线计算与 API 调用。

**MVP 边界**：不做真实支付扣款；不接入真实高德/美团线上 API，全量使用本期 Mock 数据池；不做多端实时协同。

| 聚合核销看板 | 卡片下钻 + 二维码集中展示 | 进入核销详情页，集中展示占座码 / 核销码 / 排队流水 | P2 |

### 2.3 非功能需求

- **首字渲染（TTFT）**：≤ 800ms
- **整体行程卡片生成**（含并行工具调用 + LLM 评分）：≤ 3.5s
- **并发**：MVP 白名单内测支撑 50 QPS
- **安全**：LLM 输出过敏感词过滤层（Guardrail）；儿童同行场景禁止推荐成人娱乐 / 未开发野景区标签 POI

## 三、交互与用户旅程

### 3.1 全流程阶段图

```
【1. 意图唤醒与追问】
       │
       ▼
【2. CoT 动态规划 + 草稿态画布共建】 ◄──── 用户编辑循环（删/换/插/Pin）
       │
       │ 用户点击 [锁定方案，一键履约]
       ▼
【3. 并行履约 + 进度流水线】
       │
       ▼
【4. 确定性出行看板（分享 / 地图 / 核销）】
       │
       ▼
【5. 伴随式异常感知 + 知情权重规划】
```

---

### 3.2 阶段一：意图唤醒与追问

用户输入："今天下午带老婆孩子出去玩，别太远"

系统不直接甩终版方案，而是**结构化轻量追问（最多 1 轮）**：

> "已为您锁定「家庭亲子」与「近郊出行」。出发时间大概几点？孩子更喜欢室内游乐还是户外公园？"

提供快捷标签（2点出发 / 室内 / 怕晒）供点击，减少输入负担。**不做问卷式多轮审讯**。

---

### 3.3 阶段二：CoT 动态规划 + 草稿态画布共建

#### CoT 动态思考过程展示（新增）

规划阶段 Agent 的推理过程**实时流式展示**，而非让用户面对空白等待。交互形态参考 Claude Extended Thinking 模式：

**展示形态**：
```
🤔 正在为您规划...

▼ 思考过程（可展开）
  ├─ 识别到：家庭场景，5岁孩子，怕晒，2点出发
  ├─ 约束提取：室内优先，3公里内，儿童友好必须
  ├─ 排除：户外公园（曝晒风险）、长排队场所（孩子易哭闹）
  ├─ 候选评分中：乐高体验馆 92分 / 绘本探索馆 87分 / 儿童科技馆 81分
  └─ 餐厅筛选：排除排队>20min，优先 child_seat 标签...

✅ 已为您生成方案
```

**规则**：
- 思考过程默认**折叠**，显示"查看规划思路 ▼"入口
- 用户点击可展开查看完整 CoT 推理，包括评分明细和排除理由
- 流式输出，逐步追加，不等全部推理完成再展示
- CoT 内容必须真实反映 LLM 的 `thought` 字段内容，不得伪造摘要

**Prompt 约束**：所有 LLM 节点的 `thought` 字段内容必须：
1. 先识别场景与特殊人员
2. 列出关键约束与排除项
3. 给出评分排序与选择理由
4. 最终输出前完成行程合理性自检（详见第五章）

---

草稿阶段没有任何真实预订，用户拥有完整主权。

**顶部动态摘要**（LLM 生成，防抖 1.5s 触发）：
> "当前方案以「3公里内低密度亲子游」为核心，含 1 个室内绘本馆与 1 个高分亲子餐厅，预计总耗时 3.5 小时，整体轻松。"

仅在节点替换、删除、插入后触发重新生成，纯拖拽排序不触发。

#### 草稿阶段用户操作清单

| 操作 | 触发方式 | 系统行为 |
|---|---|---|
| **删除节点** | 点击卡片右上角 [X] | 沙盒退出，前后时间轨自动平滑缩短，冲突重算 |
| **替换节点** | 点击卡片 [换一个] | 卡片原地翻转，拉起半屏：上方小地图（标注 3 个备选 POI），下方货架列表；选择后完成替换 |
| **插入节点** | 点击时间轴 [+] | 弹出输入框或标签选择（餐厅/充电桩等），AI 检索匹配 POI，精准插单并重新计算时间 |
| **延后 / 提前** | 卡片菜单 → [延后] / [提前] | 以 15min 为步长调整节点时间，后续节点联动顺延，冲突重算 |
| **跳过** | 卡片菜单 → [跳过] | 节点标记为跳过态，不参与履约，时间轴折叠该节点 |
| **标记已完成** | 卡片菜单 → [已完成] | 节点进入 completed_lock，后续 Replanning 不触碰 |
| **出问题了** | 卡片菜单 → [出问题了] | 进入半结构化冲突上报流（详见第九章） |
| **手动 Pin** | 长按卡片 → [标记高优，不要轻易换] | 节点标记 `user_pinned: true`，卡片显示 📌，Replanning 绝对保护 |
| **取消 Pin** | 再次长按 → [取消高优标记] | 恢复普通节点 |
| **模式切换** | 顶部工具栏 [全托管模式] 开关 | 切换提醒密度和时间约束强度（见第四章），仅草稿阶段可切换 |

**修改粒度控制**：单次规划会话内，同一节点替换次数上限为 5 次，超出后提示"建议重新描述您的需求，为您重新规划"。

### 3.4 阶段三：并行履约 + 进度流水线

**履约策略**：所有节点**并行发起履约请求**，但**按行程时序在前端展示状态**，建立行程感。

**状态流 UI**：
```
正在帮你安排下午～

✅ 亲子乐高体验馆    预约成功
⌛ 亲子主题餐厅      正在线上取号...
⌛ 充电桩停车位      锁单确认中...
✅ 全程路线          已生成

预计还需 28 秒  ████░░░░░░
```

| 状态 | 图标 | 含义 |
|---|---|---|
| 待履约 | 🔘 | 请求尚未发出 |
| 履约中 | ⌛ | 请求已发出，等待响应 |
| 成功 | ✅ | 接口返回成功，节点锁定 |
| 失败/降级 | ⚠️ | 接口失败，自动触发 Fallback 并提示 |

倒计时根据待处理节点数动态估算，全部完成后自动跳转看板。

### 3.5 阶段四：出行看板

履约完成后生成完整计划卡片：

- 顶部天气提示 + 全程车程/步行预估
- **地图模式（P2）**：全屏大地图，行程节点 ①②③④ 彩色气泡标注，实时路况线路；气泡点击展开 POI 简介，底部 [导航] 按钮跳转外部导航 App
- **核销看板（P2）**：点击节点进入核销详情页，集中展示占座二维码、核销码、排队流水号
- **社交分享（P0）**：一键生成精美长图导出，支持分享至微信群，群内好友可只读查看同款计划

---

### 3.6 阶段五：伴随式异常感知与重规划

**触发条件**：

| 异常类型 | 判定阈值 | 触发逻辑 |
|---|---|---|
| 排队暴增 | 等待 > 30min（全托管）/ > 45min（轻托管），且影响后续衔接 | 局部重规划未来受影响节点 |
| 天气骤变 | 气象接口返回 heavy_rain / storm | 露天节点替换为室内备选 |
| 用户疲劳 | 用户发送"太累了"/"孩子困了"等语义 | 行程压缩，推荐就近结束 |

**知情权弹窗**：系统绝不擅自修改终版方案，必须弹窗：

> "管家发现突发状况：餐厅前方突发暴增 45 桌。已为您找到同商圈、无需排队的【亲子主题西餐厅】，是否切换？"
>
> [同意切换]　[暂不处理，我自己看看]

用户同意后：completed_lock 节点与 user_pinned 节点锁定不动，仅替换受影响的 pending 节点。

## 四、模式分层系统

### 4.1 模式定义

| 维度 | 轻托管（默认） | 全托管 |
|---|---|---|
| 适用人群 | P 人聚会、随性出行 | 家庭带娃、时间紧、预约密集 |
| 时间约束 | 模糊时间块（上午/下午/傍晚），±30min 弹性 | 精确时间轨（14:00 乐高 → 16:30 餐厅），节点间留 15min 缓冲 |
| 提醒密度 | 低：仅在严重冲突或到达时提醒，不主动推送 schedule | 高：节点前 15min 推送提醒，时间冲突实时警告 |
| 超时处理 | 弹窗提示，用户选择权大，可选"暂不处理" | 弹窗更紧迫，超时无操作后系统自动执行推荐方案 |
| 行程锁定 | 软锁为主，可随时打断调整 | 强锁定，调整需明确二次确认 |
| Replanning 阈值 | 排队 > 45min 才介入 | 排队 > 30min 即介入 |
| 评分权重 | 氛围感、评分权重高；时间约束权重低 | comfort_score、child_friendly 权重高；时间精度权重高 |

### 4.2 模式切换规则

- 入口：草稿阶段顶部工具栏 [全托管模式] 开关
- **锁定履约后模式固化**，不可中途切换，避免提醒逻辑混乱
- 全托管模式下，行程卡片顶部显示橙色"托管中"角标

## 五、行程合理性校验（独立章节）

### 5.1 为什么需要独立的合理性校验

LLM 具备强大的语义理解能力，但在处理"看起来可行，实则体验极差"的方案时容易失效。典型问题是：**所有单点约束都满足，但整体体验严重翻车**。

**反面案例（高德真实 BadCase）**：
> 用户问"周末想游山玩水"，被推荐京郊青龙峡。用户说"没有车"（隐含：太远了），系统没有确认意图、没有重新推荐，而是直接优化原方案，让用户坐公交。用户说"时间不够"，系统仍未确认，继续查询公交班次和景点开门时间，最终给出"可行"结论：公交往返 6 小时，游玩时间仅 1 小时。营业时间✓、公交路线✓、物理上"来得及"✓，但通勤/游玩比例 = 6:1，这是一次彻底的体验翻车。

**核心原则**：Agent 必须在输出方案前完成合理性自检，**拒绝输出看起来合法但体验极差的方案**。同时，当用户补充信息（如"没有车"）时，必须先**确认意图**，而不是直接执行推断出的子任务。

### 5.2 合理性校验规则集

以下规则在 LLM 的 `thought` 字段中强制执行，任何一条不通过则方案不得输出，必须重新规划：

**规则一：通勤/游玩时间比例**
- 单程交通耗时不得超过总出行时长的 30%
- 往返交通耗时合计不得超过总出行时长的 40%
- 违反示例：总行程 4 小时，往返通勤 2.5 小时 → 不合格，必须换更近的目的地

**规则二：核心活动时间保障**
- 每个主要活动节点的有效游玩时间不得少于 45 分钟（不含交通）
- 含儿童场景：儿童活动时间不得少于 1 小时，避免"赶场"
- 违反示例：游玩时间仅 1 小时但通勤 6 小时 → 不合格

**规则三：交通方式与距离匹配**
- 无车用户（步行/公交）：单程距离超过 15 公里的目的地禁止推荐
- 步行出行：单程超过 2 公里禁止推荐步行方式，必须建议打车/骑行
- 含儿童：单程步行超过 800 米需标注风险

**规则四：意图确认优先于方案优化**
- 当用户的补充信息（"没有车"、"时间不够"、"太远了"）与当前方案存在根本冲突时，**必须先确认用户意图**（是否需要重新选目的地），而不是直接在原方案上优化
- 禁止行为：用户说"太远了" → 系统查公交路线而不重新推荐

**规则五：节奏合理性**
- 全天行程节点数：建议 2-4 个主活动 + 1 顿饭，最多不超过 6 个节点
- 相邻节点间必须有合理交通缓冲时间（不能规划 16:00 乐高结束，16:01 到达餐厅）
- 含儿童：行程末节点需在晚 20:00 前结束，避免晚归

**规则六：季节与天气适配**
- 雨天 / 极端高温（>37°C）/ 极端低温（<0°C）：禁止推荐长时间户外活动
- 当前天气 = heavy_rain 时，户外节点自动触发风险警告

### 5.3 合理性自检在 CoT 中的呈现

LLM 在输出方案前，`thought` 字段必须包含合理性自检段落：

```
合理性自检：
- 通勤比例：往返约 40min / 总时长 240min = 17% ✓
- 核心活动时间：乐高 90min ✓ / 餐厅 60min ✓
- 步行距离：乐高→餐厅 约 800m，建议打车提示 ⚠️
- 节点数量：3个节点 ✓
- 儿童结束时间：预计 19:30 结束 ✓
结论：方案合格，可输出
```

## 六、系统 Workflow 工作流编排

```
[用户输入 / 追加操作]
         │
         ▼
(1. LLM 意图与操作解析)  ──► [CoT 流式展示给用户]
         │
         ▼
(2. 确定性并行工具链)  ──► /api/location + /api/weather + /api/activities/search
         │
         ▼
(3. LLM 损益评分 + 行程合理性自检)  ──► [CoT 流式展示 评分过程]
         │
         ├── 合理性不通过 ──► 重新规划，禁止输出
         │
         ▼
(4. 草稿拼装渲染)  ◄──── 用户编辑循环
         │
         │ 用户点击 [一键履约]
         ▼
(5. 资源策略系统)  ──► 根据资源类型决定软锁时机
         │
         ▼
(6. 并行履约引擎)  ──► 工具调用前中后状态检查
         │
         ▼
(7. 看板生成 + 伴随监控)  ──► [突发异常] ──► (8. 局部重规划路由)
                                                        │
                                                        ▼
                                              [知情权弹窗 → 用户确认]
```

### 节点 1：LLM 意图与操作解析器

输入：用户自然语言 / 草稿阶段追加操作

识别两类信息：
1. **实体与约束**：scene / companions / time / location / preferences
2. **操作语义**：OPERATION = INSERT / DELETE / REPLACE / PIN / UNPIN / DELAY / SKIP

**意图冲突检测**：当用户补充信息与当前方案存在根本冲突时（如"没有车"对应远郊景区），解析器输出 `intent_conflict: true`，触发意图再确认而非方案优化。

输出示例：
```json
{
  "scene": "family",
  "companions": ["adult_female", "child_age_5"],
  "trip_rigidity": "high",
  "start_time": "14:00",
  "preferences": ["indoor", "avoid_sun"],
  "intent_conflict": false,
  "operation": {
    "action_type": "INSERT_NODE",
    "target_category": "EV_CHARGING",
    "insert_after_node_id": "node_001_lego",
    "constraints": { "connector_type": "FAST", "distance_limit_km": 2 }
  }
}
```

### 节点 2：确定性并行工具链

并行调用：`/api/location` / `/api/weather` / `/api/activities/search`

输出：纯事实数据池（Raw Fact Pool），不含任何 LLM 推断

### 节点 3：LLM 损益评分 + 合理性自检

输入：Raw Fact Pool + 用户约束 + 当前模式（`mode: light | full`）

必须在 `thought` 字段中完成：① 模式确认 → ② 特殊人员约束 → ③ POI 评分排序 → ④ 行程合理性自检

输出：候选 POI 排行榜 + `planner_reason` + `feasibility_check`

### 节点 4：确定性路由

判断条件：存在 `soft_locked == true` 或 `user_pinned == true` 的节点

- YES → 局部重规划器（仅重构 pending 节点，绝对保护锁定节点）
- NO → 完整行程生成器

1. 当前模式确认 + 约束阈值
2. 特殊人员识别（儿童/老人/减肥等）及对应约束
3. POI 关键损益项拆解
4. 行程合理性自检（通勤比例 / 活动时间 / 交通方式匹配）
5. 最终结论

# Output Schema
{

  "poi_id": "string",

## 八、偏好锁与资源策略系统

### 8.1 锁的类型定义

| 锁类型 | 来源 | 优先级 | 说明 |
|---|---|---|---|
| `completed_lock` | 系统自动 | 最高 | 已履约成功的节点，不可修改、不可删除 |
| `user_pinned` | 用户手动 | 次高 | 用户长按标记，Replanning 绝对保护 |
| `soft_lock` | 系统自动触发 | 普通 | 资源进入高风险窗口时自动触发，替换时必须告知用户替换风险并二次确认 |

### 8.2 资源策略系统（软锁差异化）

不同类型资源的锁定时机根本不同，不能使用同一套策略：

**景点 / 演出类（稀缺资格型）**
- 特征：门票稀缺、提前放票、取消有损失（退款难 / 资格释放）
- 软锁触发时机：**计划生成时即触发软锁**
- 软锁内容：提示用户"此景点门票较难抢，建议立即预约以锁定资格"
- 取消时触发资源损失警告（见第九章）

**餐厅类（动态时间型）**
- 特征：不稀缺、最优预约时机是出发前 1-1.5 小时（太早不如不约）
- 软锁触发时机：**动态计算**，在出发前约 90 分钟、排队预测超过阈值时触发
- 软锁内容："距离用餐约 90 分钟，现在是最佳取号时机，是否帮您取号？"
- 不应在计划生成时就软锁餐厅，否则占用排队资格又浪费

**充电桩 / 停车类（实时可用型）**
- 特征：完全实时，提前锁定无意义
- 软锁触发时机：**出发前 15 分钟**才触发
- 不需要早锁

**软锁触发公式**：`软锁优先级分 = 稀缺系数 × 替代难度 × 取消损失系数`

### 8.3 节点数据结构

```json
{
  "node_id": "node_002_restaurant",
  "poi_id": "res_001",
  "poi_name": "花园简餐 Bistro",
  "category": "restaurant",
  "scheduled_time": "18:00",
  "duration_min": 60,
  "status": "pending",
  "completed_lock": false,
  "user_pinned": false,
  "soft_lock": false,
  "soft_lock_reason": null,
  "resource_type": "restaurant",
  "conflict_acknowledged": false,

    {
      "type": "queue_time",
      "severity": "warning",
      "message": "当前排队约 35 分钟，可能影响孩子用餐时间"
    }
  ]
}
```

## 九、资源损失警告 + 用户主动上报冲突

### 9.1 资源损失警告（Resource Loss Warning）

当用户尝试取消或替换一个**已有成本的节点**（已预约、已取号、已付款、余票稀缺）时，必须在执行前弹出损失提醒：

**弹窗示例**：
> 取消当前计划后：
> - 故宫预约资格将立即释放
> - 当前时段余票已售罄，再次预约成功率极低
> - 后续可约日期仅剩 2 天
>
> 是否确认取消？
> [确认取消]　[保留，我再想想]

**触发条件**：
- 节点状态为 soft_locked 或 completed_lock
- 节点存在已付款订单
- 节点 resource_type = "attraction" 且稀缺系数 > 0.7
- 排队资格已占用（已取号）

**禁止行为**：在未展示损失警告的情况下直接执行取消或替换。

### 9.2 用户主动上报冲突（半结构化流）

用户点击卡片 [出问题了] 后，进入以下交互流程：

**第一步：多选原因（系统提供选项，减少输入负担）**
```
遇到什么问题了？（可多选）
☐ 排队太久了
☐ 活动时间超了
☐ 下雨 / 天气不好
☐ 孩子/朋友不想去了
☐ 地方找不到 / 路线有问题
☐ 其他（自己说）
```

**第二步**：用户选择"其他"时，弹出自然语言输入框，用户用文字描述问题。

**第三步**：系统识别冲突类型，给出 2-4 个恢复方案（按推荐顺序排列），用户选择一个执行：
```
根据您的情况，为您准备了以下方案：

① 换【同商圈亲子主题餐厅】，当前排队 5 分钟，步行 3 分钟可达  ⭐推荐
② 延后本节点 30 分钟，当前排队预计届时减少到 15 分钟
③ 跳过本节点，直接前往下一站

[选择方案①]　[选择方案②]　[选择方案③]　[都不合适，我自己调]
```

**禁止行为**：选项不超过 4 个，不给用户"无限可能"，减少选择负担。

## 十、冲突可视化系统

### 10.1 卡片状态三级分层

| 卡片状态 | 视觉样式 | 触发条件 | 用户操作 |
|---|---|---|---|
| **正常卡** | 白色/默认 | 无冲突，无风险 | 标准操作 |
| **风险卡** | 黄色边框 + ⚠️ | 非阻断性冲突（距离偏远、排队较长、儿童不适配） | 可忽略，可处理 |
| **严重冲突卡** | 红色边框 + ❗ + 卡片展开 | 阻断性冲突（时间逆序、营业时间冲突、物理不可达） | 必须处理，履约按钮置灰 |

严重冲突卡自动展开，展示：
- 冲突描述（"到达餐厅预计 19:10，但营业截止 19:00"）
- 推荐恢复操作（[换一个] / [调整时间]）

### 10.2 冲突类型与展示规则

| 冲突类型 | 检测逻辑 | 卡片状态 |
|---|---|---|
| 时间冲突 | 相邻节点含交通总时长超出可用窗口 | 严重冲突卡 🔴 |
| 营业时间冲突 | 预计到达时间晚于 POI 营业截止 | 严重冲突卡 🔴 |
| 通勤比例超标 | 往返通勤 > 总时长 40% | 严重冲突卡 🔴 |
| 距离偏远 | 步行 > 2km 但出行方式为步行 | 风险卡 🟡 |
| 排队较长 | 排队 > 轻托管 45min / 全托管 30min | 风险卡 🟡 |
| 儿童不适配 | POI 无 child_friendly，companions 含儿童 | 风险卡 🟡 |

冲突警告内联渲染在卡片上，不弹窗打断。用户可点击 [忽略]（记录 `conflict_acknowledged: true`），系统不阻断但在核销看板中持续提示。

## 十一、安全机制

### 11.1 高风险动作拦截

系统识别并**强制拦截**以下规划内容，不得输出给用户：

| 风险类型 | 拦截规则 |
|---|---|
| 儿童不适场所 | companions 含儿童时，禁止推荐：酒吧、夜店、成人娱乐、评级 R 场所 |
| 高强度活动 | 含儿童 / 老人时，禁止推荐：高强度徒步（>5km）、极限运动 |
| 极端天气户外 | 大雨 / 极端高温低温时，禁止推荐露天长时间活动 |
| 晚归风险 | 含儿童时，行程末节点超过 20:00 触发警告 |
| 酒驾风险 | 行程含饮酒场所（酒吧/小酒馆）且用户有驾车回程时，提示代驾 |
| 不安全路线 | 禁止推荐需要穿越非开放区域的路线 |

### 11.2 行程物理合理性强制校验

在履约前，系统必须完成以下校验，**任何一项不通过则禁止进入履约阶段**：

- 时间不逆序（node[i].end_time < node[i+1].start_time）
- 相邻节点间交通耗时已纳入时间轴
- 没有推荐已确认关闭的 POI（closed_today == true）
- 通勤/游玩比例校验通过（见第五章）

### 11.3 工具调用安全

所有外部 Mock 接口调用遵循：
- **调用前**：确认是否需要用户二次确认（涉及真实预约/取号）
- **调用中**：监控响应状态，超时 2s 触发重试
- **调用后**：状态同步至节点，失败状态必须显式告知用户

**禁止行为**：工具调用失败时伪装为成功状态展示给用户。

### 11.4 用户确认机制

凡是涉及真实世界成本的动作，必须二次确认，**不允许 AI 擅自执行**：

| 动作类型 | 是否需要二次确认 |
|---|---|
| 预约景点 / 取门票 | ✅ 必须确认 |
| 餐厅取号 | ✅ 必须确认 |
| 取消已预约节点 | ✅ 必须确认 + 损失警告 |
| 替换已 soft_lock 节点 | ✅ 必须确认 + 损失警告 |
| 添加新节点（无成本） | 不需要 |
| 草稿阶段替换（无成本） | 不需要 |

### 11.5 隐私安全

- 用户位置、家庭成员信息、儿童年龄、支付信息：最小化采集，用户可删除，脱敏展示
- 分享计划时：仅暴露行程内容，不暴露下单信息、付款金额、个人联系方式

## 十二、Fallback 分层体系

### 12.1 Fallback 五级策略

| 级别 | 策略 | 说明 |
|---|---|---|
| Level 0 | 重试 | 接口失败重试 1 次 |
| Level 1 | 同类替换 | 同类型 POI 替换（火锅 → 烤肉） |
| Level 2 | 同商圈替换 | 同商圈内跨类型替换（火锅 → 简餐） |
| Level 3 | 删减非关键节点 | 跳过可选活动，保留主活动 + 餐饮 |
| Level 4 | 重规划剩余行程 | 基于当前位置和剩余时间全量重规划 |
| Level 5 | 人工处理提示 | 系统无法自动解决，明确告知用户手动处理 |

### 12.2 十类失败场景处理

**场景 1：AI 生成失败**（大模型无返回 / JSON 解析失败）
- 重试 1 次 → 仍失败 → 使用规则模板生成基础行程（附近活动 → 餐厅 → 回家）

**场景 2：POI 匹配失败**（搜索无结果）
- 扩大搜索范围 → 换关键词（亲子乐园 → 儿童乐园 / 商场儿童区）→ 推荐同类型备选

**场景 3：餐厅不可用**（满座 / 排队过长 / 暂停营业）
- 同类替换 → 同商圈替换 → 降低复杂度（改为轻食快餐）

**场景 4：预约/下单失败**（接口失败 / 余票不足）
- 重试 → 换时段 → 换同类 POI → 提示用户手动处理
- **严禁伪装成功**

**场景 5：执行超时**（当前活动超时，后续来不及）
- 压缩后续活动 → 跳过非关键节点 → 延后餐饮/预约 → 重规划剩余行程

**场景 6：用户临时变更**（孩子累了 / 朋友想换地方）
- 保留 completed_lock 节点 → 保护 user_pinned 节点 → 只重规划 pending 节点

**场景 7：软锁节点被取消**（用户想取消有成本节点）
- 弹出资源损失警告 → 用户确认后才执行 → 取消后自动重规划后续

**场景 8：定位/地图失败**
- 使用用户手动输入位置 → 商圈级推荐 → 不计算精确路线，给附近方案

**场景 9：全局计划崩溃**（下雨 + 多节点不可用 + 晚出发）
- 触发全局恢复 → 重规划剩余时间 → 生成低风险室内兜底方案

**场景 10：Mock 数据池备选穷尽**
- 展示"附近符合条件的备选已用完" → 提供 [扩大到 5 公里] / [换个类型] 快捷选项
- 不允许 LLM 编造不存在的 POI

## 十三、训练数据与 Case 库

### 13.1 Good/Bad Case 库（扩充版）

---

**[场景一] 家庭带娃 + 餐厅排队过长**

✅ Good Case：
- Thought：用户带 5 岁孩子，"轻食研究所"突发排队 90 分钟，孩子极易因饥饿哭闹。启动 Fallback，completed_lock 乐园节点，同商圈 500 米内替换为排队 < 20 分钟且 child_friendly 的"花园简餐 Bistro"，合理性自检通过。
- Output：锁定 node_001，node_002 替换为 res_002

❌ Bad Case A（全盘推翻）：因餐厅排队，连带删除用户已完成的乐园 → 违反 completed_lock
❌ Bad Case B（幻觉编造）：编造 Mock 池不存在的餐厅 → 违反数据作用域限制
❌ Bad Case C（User Pin 破坏）：用户 Pin 了某餐厅，Replanning 仍替换 → 违反 user_pinned 保护

---

**[场景二] 朋友聚会 + 天气骤变（暴雨）**

✅ Good Case：检测到 heavy_rain，仅将露天 Citywalk 替换为室内艺术展，维持傍晚已预约的室内小酒馆不变

❌ Bad Case：将不受天气影响的室内小酒馆也取消 → 违反"只改受影响未来节点"原则

---

**[场景三] 通勤/游玩比例翻车**（对应第五章 BadCase）

✅ Good Case：
- 用户说"周末想游山玩水，没有车"
- Thought：无车用户，识别到"没有车"是对"太远"的隐含表达，检测到当前方案（京郊景区）单程公共交通 3 小时，往返 6 小时，游玩时间 1 小时，通勤/游玩比例 = 6:1，严重不合格。必须先确认用户意图：是否换一个市区内可到达的目的地？
- Action：输出意图确认提问："京郊景区坐公交往返需要约 6 小时，游玩时间只剩 1 小时，建议换到市区内的景点，比如城市公园或商圈附近的活动，是否重新推荐？"
- Output：待用户确认后，推荐通勤 < 1 小时的市区内目的地

❌ Bad Case A（意图错位）：用户说"没有车"→ 系统不确认意图，直接查询公交路线 → 给出"可行"结论但通勤/游玩比例 = 6:1 → 严重翻车
❌ Bad Case B（自欺欺人）：方案中单程交通 3 小时，系统只校验了营业时间和路线存在，未校验通勤比例，标记方案合格输出

---

**[场景四] 特殊人员约束被忽略**

✅ Good Case：
- 用户说"带减肥的老婆出去吃饭"，系统识别 special_person: 减肥者，在餐厅候选中过滤掉高油高糖餐厅，优先推荐轻食、沙拉、低卡菜系
- 老婆约束优先级低于儿童但高于普通成人

❌ Bad Case：用户提到"老婆在减肥"，系统忽略该约束，推荐了火锅/烧烤 → 违反特殊人员约束提取规则

---

**[场景五] 软锁替换未告知损失**

✅ Good Case：
- 用户要取消已取号的热门餐厅
- 系统弹出："当前餐厅已为您取号，前方还有 8 桌，预计 35 分钟后轮到您。取消后排队资格立即释放，若想再来需重新排队。是否确认取消？"
- 用户确认后才执行取消

❌ Bad Case：系统直接取消已取号的节点，未告知用户排队资格损失 → 违反资源损失警告规则

---

**[场景六] 用户意图追问时机错误**

✅ Good Case：
- 用户第一轮只说"出去玩"，系统追问出发时间和偏好，**只问 1 轮**
- 用户回答后生成方案，不再追问

❌ Bad Case：系统进行 3 轮以上追问（"您的预算是多少？""您喜欢什么风格？""您对噪音敏感吗？"），像问卷一样 → 决策成本极高，用户放弃

## 十四、Mock 数据边界

### 14.1 Mock 数据池规格

| 分类 | 最小数量 | 必含字段 |
|---|---|---|
| 亲子场馆 | ≥ 8 个 | child_friendly, child_seat, queue_time_min |
| 餐厅（含亲子主题） | ≥ 12 个 | child_seat, open_hours, queue_time_min, diet_tags |
| 室内娱乐/艺术展 | ≥ 6 个 | indoor: true, open_hours |
| 充电桩/停车场 | ≥ 5 个 | connector_type, availability |
| 户外公园 | ≥ 4 个 | outdoor: true, open_hours |

每个 POI 必须包含：`poi_id, name, category, location, open_hours, queue_time_min, child_friendly, distance_km, rating, tags, closed_today`

### 14.2 边界与 Fallback

- **换一换穷尽**：提示"附近符合条件的备选已用完"，提供 [扩大范围] / [换个类型]
- **Replanning 无解**：不执行替换，弹窗"当前无合适备选，请手动调整"，保留原节点标记 `fallback_failed: true`
- **接口超时（>2s）**：自动重试 1 次，重试失败标记"履约失败"，提示用户现场处理
- **禁止 LLM 编造不在 Mock 池中的 POI**，违反即输出无效

## 十五、评测体系

### 15.1 评测维度（Benchmark-300）

| 维度 | 指标 | 目标值 |
|---|---|---|
| 语义解析 | scene / trip_rigidity / 特殊人员约束对齐率 | ≥ 95% |
| 逻辑冲突率 | 时间逆序 / 距离超限 / 推荐闭馆 POI | = 0% |
| 合理性校验通过率 | 通勤比例 / 活动时间 / 交通匹配规则 | = 0% 漏出不合格方案 |
| 锁保护率 | completed_lock / user_pinned 节点被擅改率 | = 0% |
| 数据作用域违规率 | 幻觉 POI 出现率 | = 0% |
| 意图确认率 | 用户补充信息与方案根本冲突时，触发意图确认而非直接执行率 | ≥ 90% |
| 模式权重差异正确率 | 轻/全托管评分差异是否符合规则 | ≥ 90% |
| 冲突恢复合理性 | LLM Judge 评审：恢复方案是否合理 | ≥ 85% |
| 损失警告触发率 | 有成本节点取消时，损失警告弹出率 | = 100% |

### 15.2 评测架构

```
用户输入样本（300 条）
       │
       ▼
AI 规划生成
       │
       ▼
规则评测  ──►  逻辑冲突 / 锁保护 / 合理性 / 数据作用域
       │
LLM Judge ──►  ① 方案是否符合用户需求？
               ② 恢复方案是否合理？
               ③ 风险提示是否清楚？
               ④ 是否擅自替用户做有成本决策？
       │
冲突模拟器 ──►  定位受影响节点 → 保护锁定节点 → 给出恢复方案 → 提醒风险
       │
       ▼
评分报告与优化建议
```

### 15.3 CI/CD 评测门禁

每次 Prompt 变更 → 自动运行 Benchmark-300 → 新版本所有指标 ≥ 旧版本 → 才允许灰度发布

### 15.4 BadCase 逆向捕获

白名单内测 → 用户点击"撤销方案"或频繁手动替换 → 完整 Context 日志自动标记 BadCase → AI PM 每周五清洗转化为新 TestCase → 充实评测集闭环
