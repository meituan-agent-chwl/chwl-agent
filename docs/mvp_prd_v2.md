# 本地生活短时活动规划 Agent — MVP PRD v2.0

> 修订说明：本版本在原文档基础上，补全了模式分层行为规则与 Prompt 联动、偏好锁机制、履约状态流顺序策略、冲突可视化、Mock 边界与 Fallback 链路五处结构性缺口，并压缩了背景介绍类内容。

---

## 一、业务背景（精简）

**核心痛点**：用户在 3-4 小时短时出行场景中，需要在多 App 之间手动串联位置、排队、天气、门票信息，决策成本极高，大量长尾出行需求在决策阶段直接流失。

**为什么用 AI**：用户意图高度模糊（"带孩子别太折腾"），传统规则引擎无法处理；现实动态不确定（排队暴增、突发暴雨），需要 Planning → Reflection → Replanning 闭环。

**核心分工**：LLM 做意图解析与动态损益评分，Python/Go 做确定性路线计算与 API 调用。

**MVP 边界**：不做真实支付扣款；不接入真实高德/美团线上 API，全量使用本期 Mock 数据池；不做多端实时协同。

## 二、目标用户与需求清单

### 2.1 用户画像

| 人群 | 场景特征 | 核心诉求 |
|---|---|---|
| 家庭带娃（J 人倾向） | 父母 + 低龄儿童，容错率极低 | 省心、低风险、反排队 |
| 朋友聚会（P 人倾向） | 3-5 人年轻群体，无目的决策 | 要氛围、要社交、无感托管 |

### 2.2 需求清单

| 模块 | 子项 | 描述 | 优先级 |
|---|---|---|---|
| 意图理解 | 语义约束提取 | 解析时间、人群、距离、隐含偏好 → 结构化标签 | P0 |
| 模式系统 | 轻托管/全托管双模 | 默认轻托管，支持切换全托管；定义各模式行为规则 | P0 |
| 混合交互 | GUI 货架微调 | 换一个/删除/插单，草稿态沙盒操作 | P0 |
| 偏好锁 | 用户主动标记高优节点 | 手动 Pin，Replanning 不得擅自改动 | P0 |
| 异常处理 | 分层 Replanning | 排队暴增/大雨/用户疲劳触发局部重构，知情权弹窗 | P0 |
| 履约状态流 | 并行履约 + 进度可视 | 按行程时序展示各节点履约状态，粗倒计时 | P1 |
| 冲突可视化 | 草稿阶段冲突提示 | 时间/距离/营业时间冲突内联警告 | P1 |
| 核销看板 | 聚合核销 + 地图联动 | 一级卡片下钻，集中展示二维码、排队流水、POI 导航入口 | P1 |

### 2.3 非功能需求

- **首字渲染（TTFT）**：≤ 800ms
- **整体行程卡片生成**（含并行工具调用 + LLM 评分）：≤ 3.5s
- **并发**：MVP 白名单内测支撑 50 QPS
- **安全**：LLM 输出过敏感词过滤层（Guardrail）；儿童同行场景禁止推荐成人娱乐/未开发野景区标签 POI

## 三、交互与用户旅程（核心重构）

### 3.1 全流程阶段图

```
【1. 意图唤醒与追问】
       │
       ▼
【2. 草稿态画布共建】 ◄──── 用户编辑（删/换/插/Pin）循环
       │
       │ 用户点击 [锁定方案，一键履约]
       ▼
【3. 并行履约 + 进度流水线】
       │
       ▼
【4. 确定性出行看板（地图 + 核销）】
       │
       ▼
【5. 伴随式异常感知 + 知情权重规划】
```

---

### 3.2 阶段一：意图唤醒与追问

用户输入："今天下午带老婆孩子出去玩，别太远"

系统不直接甩终版方案，而是**结构化轻量追问**：

> "已为您锁定「家庭亲子」与「近郊出行」。出发时间大概几点？孩子更喜欢室内游乐还是户外公园？"

提供快捷标签（2点出发 / 室内 / 怕晒）供点击，减少输入负担。追问轮次：**最多 1 轮**，不做问卷式多轮审讯。

---

### 3.3 阶段二：草稿态画布共建（核心）

草稿阶段没有任何真实预订，用户拥有完整主权。

**顶部动态摘要**（LLM 生成）：
> "当前方案以「3公里内低密度亲子游」为核心，含 1 个室内绘本馆与 1 个高分亲子餐厅，预计总耗时 3.5 小时，整体轻松。"

**摘要刷新策略**：防抖处理，用户操作停止后 1.5s 触发重新生成，避免每次微调都调用 LLM。仅在以下操作后触发：节点替换、节点删除、插入新节点。纯拖拽排序不触发。

#### 草稿阶段用户操作清单

| 操作 | 触发方式 | 系统行为 |
|---|---|---|
| **删除节点** | 点击卡片右上角 [X] | 节点沙盒退出，前后时间轨自动平滑缩短，冲突重算 |
| **替换节点** | 点击卡片 [换一个] | 卡片原地翻转，拉起半屏：上方小地图（多点标注 3 个备选 POI），下方货架列表；用户选择后完成局部替换 |
| **插入节点** | 点击时间轴 [+] 按钮 | 弹出输入框或标签选择（餐厅/充电桩/便利店等），AI 在两节点间检索匹配 POI，精准插单并重新计算时间 |
| **手动 Pin（偏好锁）** | 长按卡片 → [标记为高优，不要轻易换] | 节点被标记 `user_pinned: true`，卡片右上角显示 📌 图标，Replanning 中该节点受保护 |
| **取消 Pin** | 再次长按 → [取消高优标记] | 恢复普通节点 |
| **模式切换** | 顶部工具栏 [全托管模式] 开关 | 实时切换提醒密度和时间约束强度（见第四章） |

#### 冲突可视化规则（详见第五章）

草稿阶段实时检测冲突，内联展示在行程轴上，不弹窗打断用户。

---

### 3.4 阶段三：并行履约 + 进度流水线

用户点击底部 **[锁定方案，一键生成计划]** 后：

**履约策略**：所有节点**并行发起履约请求**（节省时间），但**按行程时序顺序在前端展示状态**，让用户建立"行程感"而非"任务列表感"。

**状态流 UI 示例**：

```
正在帮你安排下午～

✅ 亲子乐高体验馆  预约成功
⌛ 亲子主题餐厅    正在线上取号...
⌛ 充电桩停车位    锁单确认中...
✅ 全程路线        已生成

预计还需 28 秒 ████░░░░░░
```

**状态节点定义**：

| 状态 | 图标 | 含义 |
|---|---|---|
| 待履约 | 🔘 | 请求尚未发出 |
| 履约中 | ⌛ | 请求已发出，等待 Mock 接口响应 |
| 成功 | ✅ | 接口返回成功，节点锁定 |
| 失败/降级 | ⚠️ | 接口返回失败，自动触发 Fallback 并提示 |

**倒计时逻辑**：根据待处理节点数动态估算（每个节点 Mock 延迟 0.5-2s），展示粗粒度倒计时进度条，全部完成后自动跳转看板。

---

### 3.5 阶段四：确定性出行看板

履约完成后生成完整计划卡片，包含：

- 顶部：天气提示 + 全程车程/步行预估
- **地图模式**：切换为全屏大地图，行程节点以 ①②③④ 彩色气泡标注，实时路况线路图。每个气泡点击展开 POI 简介，底部提供 **[导航]** 按钮，调用美团/高德地图 App（跳转外部导航）
- **核销看板（卡片下钻）**：点击任意节点进入核销详情页，集中展示占座二维码、核销码、排队流水号（前方还有 X 桌）。出行中无需切换其他 App
- **分享**：一键生成长图导出，支持分享至微信群，群内好友可以只读或共同编辑模式查看

---

### 3.6 阶段五：伴随式异常感知与重规划

**触发条件**（三类异常）：

| 异常类型 | 判定阈值 | 触发逻辑 |
|---|---|---|
| 排队暴增 | 预计等待时间 > 30min，且影响后续节点衔接 | 局部重规划未来受影响节点 |
| 天气骤变 | 气象接口返回 heavy_rain / storm | 将露天节点替换为室内备选 |
| 用户疲劳信号 | 用户主动发送"太累了"/"孩子困了"等语义 | 触发行程压缩，推荐就近结束或缩短节点 |

**知情权弹窗原则**：系统**绝不擅自修改已确认的终版方案**，必须弹窗告知：

> "管家发现突发状况：您预约的餐厅前方突发暴增 45 桌。为不耽误孩子吃饭，已为您找到同商圈、无需排队的【亲子主题西餐厅】，是否切换？"
>
> [同意切换]  [暂不处理，我自己看看]

用户点击 [同意切换] 后：历史已完成节点与 user_pinned 节点**牢牢锁定**，仅替换受影响的未来节点。

## 四、模式分层系统（新增章节）

### 4.1 模式定义

| 维度 | 轻托管模式（默认） | 全托管模式 |
|---|---|---|
| **适用人群** | P 人聚会、随性出行 | 家庭带娃、时间紧、预约密集 |
| **时间约束** | 模糊时间块（"上午 / 下午 / 傍晚"） | 精确时间轨（14:00 乐高 → 16:30 餐厅） |
| **提醒密度** | 低：仅在严重冲突或到达时提醒，不主动推送 schedule | 高：每个节点提前 15min 推送提醒，时间冲突实时警告 |
| **异常处理** | 弹窗提示，用户选择权更大，可选"暂不处理" | 弹窗更紧迫，超时无操作后系统自动执行推荐方案 |
| **行程锁定强度** | 软锁为主，可随时打断调整 | 强锁定，调整需要明确二次确认 |
| **Replanning 触发阈值** | 排队 > 45min 才介入 | 排队 > 30min 即介入 |

### 4.2 模式切换

- **入口**：草稿阶段顶部工具栏 [全托管模式] 开关，默认关闭（轻托管）
- **切换时机**：仅在草稿阶段可自由切换；锁定履约后，模式固化，不可中途切换（避免提醒逻辑混乱）
- **视觉区分**：全托管模式时，行程卡片顶部显示橙色"托管中"角标

### 4.3 模式与 Prompt 联动规则（详见第六章）

两套模式在 LLM 评分阶段使用不同的权重配置和输出约束，不能共享同一套评分 Prompt。

## 五、冲突可视化系统（新增章节）

### 5.1 冲突类型与检测时机

| 冲突类型 | 检测逻辑 | 展示层级 |
|---|---|---|
| **时间冲突** | 相邻两节点（含交通耗时）总时长超出可用窗口 | 🔴 红色警告，阻断性 |
| **距离冲突** | 两节点之间步行距离 > 2km 但用户出行方式选了"步行" | 🟡 黄色提示，非阻断性 |
| **营业时间冲突** | 节点预计到达时间晚于 POI 的营业截止时间 | 🔴 红色警告，阻断性 |
| **儿童不适配** | POI 无 child_friendly 标签，但 companions 包含儿童 | 🟡 黄色提示，非阻断性 |

### 5.2 展示规则

- **内联展示**：冲突警告直接渲染在行程轴对应节点卡片上，不弹窗打断
- **样式**：卡片底部出现彩色条 + 简短说明，例如：
  - 🔴 `时间冲突：到达餐厅预计 19:10，但营业截止 19:00`
  - 🟡 `距离较远：从乐高到餐厅步行约 25 分钟，是否改为打车？`
- **用户处理选项（内联）**：冲突条旁提供两个快捷按钮 [换一个] / [忽略] ，用户可强行保留冲突节点（系统记录但不阻断）
- **阻断性冲突强制处理**：红色冲突存在时，[锁定方案，一键履约] 按钮置灰，必须处理后才可进入履约阶段

### 5.3 用户强制保留冲突节点

用户点击 [忽略] 后，节点被标记 `conflict_acknowledged: true`，行程可正常进入履约，但该节点在核销看板中显示 ⚠️ 提示："此节点存在您已知晓的冲突，请注意现场情况。"

## 六、系统 Workflow 工作流编排

系统采用**状态机循环流**，区分草稿态（Draft Canvas）与履约态（Fulfillment Lock）。

```
[用户输入 / 追加操作] ──► (1. 语义与操作解析)
                                  │
                                  ▼
                          (2. 事实池并发捞取)
                                  │
                                  ▼
                          (3. 损益评分与草稿拼装) ◄── 用户编辑循环
                                  │
                         用户点击 [一键履约]
                                  │
                                  ▼
                          (4. 并行履约引擎)
                                  │
                                  ▼
                          (5. 看板生成与伴随监控) ──► [突发异常]
                                                              │
                                                              ▼
                                                    (6. 局部重规划路由)
                                                              │
                                                              ▼
                                                    [知情权弹窗 → 用户确认]
```

### 节点 1：LLM 意图与操作解析器

输入：用户自然语言，或草稿阶段追加操作（"中间插个充电桩"）

解析器识别两类信息：
1. **实体与约束**：scene / companions / time / location / preferences
2. **操作语义**：OPERATION = INSERT / DELETE / REPLACE / PIN / UNPIN

输出示例：

```json
{
  "scene": "family",
  "companions": ["adult_female", "child_age_5"],
  "trip_rigidity": "high",
  "start_time": "14:00",
  "preferences": ["indoor", "avoid_sun"],
  "operation": {
    "action_type": "INSERT_NODE",
    "target_category": "EV_CHARGING",
    "insert_after_node_id": "node_001_lego",
    "constraints": { "connector_type": "FAST", "distance_limit_km": 2 }
  }
}
```

### 节点 2：确定性并行工具链

并行调用：`/api/location` / `/api/weather` / `/api/activities/search`

输出：纯事实数据池（Raw Fact Pool），不含任何 LLM 推断

### 节点 3：LLM 损益推理与评分器

输入：Raw Fact Pool + 用户约束 + 当前模式（`mode: light | full`）

执行 COT 推理，计算 6 维权重分值，过滤不适 POI

**评分权重因模式不同而差异化**（详见第七章 Prompt 规范）

输出：候选 POI 排行榜 + `planner_reason`

### 节点 4：确定性路由

判断条件：存在 `soft_locked == true` 或 `user_pinned == true` 的节点

- **YES**：路由至局部重规划器（仅重构未来，注入约束冲突校验，user_pinned 节点受绝对保护）
- **NO**：路由至完整行程生成器

## 七、大模型选型与 Prompt 规范

### 7.1 模型选型

| 模型 | 结构化输出 | 延迟 | 选型结论 |
|---|---|---|---|
| Qwen-2.5-14B-Instruct | 良好 | ~900ms | **MVP 主力**，性价比最优 |
| DeepSeek-V3 | 优秀 | ~1.5s | 高配备用，复杂 Replanning 场景升级使用 |
| Qwen-2.5-7B-Instruct | 偶发 JSON 破碎 | ~600ms | **淘汰**，多维评分顾此失彼 |

选型原则：从小参数开始验证，效果相似优先小参数。

### 7.2 Prompt 工程规范

- 所有 Prompt 用 Git 独立版本管理，变更必须经过评测集跑分验证
- 解析、评分、重规划分属三个独立 Prompt 模板，避免单 Prompt 过长衰减
- 强制 JSON Schema 输出，不合规直接拦截并重试（Max Retry = 2）

### 7.3 核心评分器 Prompt（`/api/candidates/score`）

```markdown
# Role
你是一个冷静、克制的城市生活管家。根据消费场景对 POI 事实数据进行严苛的损益评估。

# 模式参数注入
当前模式: {mode}  # "light_managed" 或 "full_managed"

# Output Constraints
1. 必须使用指定 JSON 结构体输出，严禁前导词、后缀或 Markdown 符号。
2. `planner_reason` 必须一句话讲清核心损益，禁止使用推销话术。

# 模式差异化评分规则

## 轻托管模式 (light_managed)
- 时间约束宽松：允许±30分钟弹性
- 排队惩罚阈值：queue_time_min > 45min 才扣分
- 不强制插入提醒节点
- 优先推荐氛围感强、评价高的 POI

## 全托管模式 (full_managed)
- 时间约束严格：节点与节点间必须留15分钟缓冲
- 排队惩罚阈值：queue_time_min > 30min 即重度扣分，comfort_score 直接 -30
- 家庭场景：必须检查 child_friendly 标签，缺失则 preference_match 折半
- 强制验证相邻节点步行距离是否在2km以内

# Chain-of-Thought 要求
必须先在 `thought` 字段中完成以下推理后才能打分：
1. 当前模式是什么，约束阈值是什么
2. 用户的隐式情绪（嫌麻烦/怕累/要氛围）
3. 该 POI 的关键损益项是什么

# Output Schema
{
  "thought": "模式确认 → 用户情绪拆解 → POI 损益比对",
  "poi_id": "string",
  "score": integer,
  "score_detail": {
    "distance_score": int,
    "queue_penalty": int,
    "preference_match": int,
    "comfort_score": int,
    "mode_adjustment": int   // 新增：模式带来的权重调整项
  },
  "planner_reason": "客观扼要的理由，不超过30字"
}
```

### 7.4 重规划器 Prompt（`/api/replan/partial`）

```markdown
# Role
你是行程重规划器。仅对受影响的未来节点进行局部替换，绝不触碰已完成节点或用户 Pin 节点。

# 硬性保护规则（违反即输出无效）
1. node.status == "completed" 的节点：绝对禁止修改或删除
2. node.user_pinned == true 的节点：绝对禁止修改或删除，即使存在冲突
3. 只允许修改 node.status == "pending" 且 node.user_pinned != true 的未来节点

# 数据作用域限制
候选替换 POI 必须且只能来自 raw_fact_pool 中的现有数据，严禁编造不存在的 POI 或 poi_id

# 约束冲突校验（替换后必须验证）
- 替换后的节点时间不得早于前序节点结束时间 + 交通耗时
- 营业时间必须覆盖预计到达时间
- 距离限制：新节点到前序节点距离须在 {distance_limit_km} 公里内

# Output Schema
{
  "thought": "触发原因 → 受保护节点确认 → 替换方案推理",
  "replanned_nodes": [
    {
      "node_id": "string",
      "action": "REPLACE | REMOVE",
      "new_poi_id": "string | null",
      "reason": "string"
    }
  ],
  "lock_violations": []   // 若为空表示无锁破坏，有内容则整体输出无效
}
```

## 八、偏好锁机制（新增章节）

### 8.1 锁的类型定义

| 锁类型 | 来源 | 优先级 | 说明 |
|---|---|---|---|
| `completed_lock` | 系统自动 | 最高 | 已履约成功的节点，不可修改、不可删除 |
| `user_pinned` | 用户手动 | 次高 | 用户长按标记，Replanning 受绝对保护 |
| `soft_lock` | 系统推断 | 普通 | 用户明确表达"这个我很想去"等语义时系统自动建议，需用户确认 |

### 8.2 数据结构

节点（Node）数据结构中新增字段：

```json
{
  "node_id": "node_002_restaurant",
  "poi_id": "res_001",
  "status": "pending",
  "completed_lock": false,
  "user_pinned": true,         // 用户手动 Pin
  "soft_lock": false,
  "conflict_acknowledged": false
}
```

### 8.3 Replanning 保护规则

重规划器（节点 4 路由 → 节点 5）在生成替换方案前，必须先过滤保护名单：

```python
protected_nodes = [
  n for n in nodes
  if n.completed_lock == True or n.user_pinned == True
]
# protected_nodes 中的任何节点不得出现在 replanned_nodes 的 action 中
```

如果 Replanning 逻辑在数学上无法在不触碰 user_pinned 节点的情况下解决冲突，系统**不执行重规划**，改为弹窗告知用户："由于您锁定了【XXX】，当前冲突无法自动解决，请手动调整。"

## 九、Mock 数据边界与 Fallback 链路（新增章节）

### 9.1 Mock 数据池规格

MVP 阶段 Mock 数据池应包含：

| 分类 | Mock POI 数量 | 覆盖区域 |
|---|---|---|
| 亲子场馆 | ≥ 8 个 | 2-3 个商圈 |
| 餐厅（含亲子主题） | ≥ 12 个 | 覆盖上述商圈 |
| 室内娱乐/艺术展 | ≥ 6 个 | 覆盖上述商圈 |
| 充电桩/停车场 | ≥ 5 个 | 覆盖上述商圈 |
| 户外公园 | ≥ 4 个 | 同区域 |

每个 POI 必须包含字段：`poi_id, name, category, location, open_hours, queue_time_min, child_friendly, distance_km, rating, tags`

### 9.2 Fallback 链路

**场景 A：换一换时备选穷尽**

当用户点击 [换一个]，系统在 Mock 池中筛选后备选 POI 数量为 0 时：
- 前端展示："附近符合条件的备选已用完，要不要放宽距离范围？"
- 提供 [扩大到 5 公里] / [换个类型] 两个快捷选项
- 不允许 LLM 编造不存在的 POI

**场景 B：Replanning 时替换方案穷尽**

重规划器在受约束条件下（同商圈 / 同类型 / 营业中 / 排队<阈值）找不到任何可用备选时：
- 不执行替换，弹窗告知用户："当前商圈内暂无合适备选，建议您手动选择或接受现有方案。"
- 保留原有节点（标记 `fallback_failed: true`），用户自行决策

**场景 C：Mock API 接口超时**

单个 Mock 接口超过 2s 未响应时：
- 该节点标记 ⚠️ 履约中，后台自动重试 1 次
- 重试失败后标记为 "履约失败"，前端展示："【XXX】预约未成功，建议到场后现场取号"
- 不影响其他节点履约状态

## 十、训练数据与 Case 库

### 10.1 Good/Bad Case 库

**[场景一] 家庭带娃 + 餐厅排队过长**

✅ Good Case：
- LLM Thought：用户带 5 岁孩子，下一节点"轻食研究所"突发排队 90 分钟。孩子极易因饥饿哭闹。启动 Fallback，锁定已完成乐园，在同商圈 500 米内替换为排队 < 20 分钟且 child_friendly 的"花园简餐 Bistro"
- Output：锁定 node_001，将 node_002 替换为 res_002

❌ Bad Case A（全盘推翻）：因餐厅排队，连带删除用户已完成的乐园节点 → 违反 completed_lock
❌ Bad Case B（幻觉编造）：编造 Mock 池不存在的餐厅 → 违反数据作用域限制
❌ Bad Case C（User Pin 破坏）：用户 Pin 了某餐厅，Replanning 仍将其替换 → 违反 user_pinned 保护

**[场景二] 朋友聚会 + 天气骤变（暴雨）**

✅ Good Case：检测到 heavy_rain，仅将露天 Citywalk 替换为室内艺术展，维持傍晚已预约的室内小酒馆不变

❌ Bad Case：将不受天气影响的室内小酒馆也取消 → 违反"只改受影响未来节点"原则

## 十一、评测体系

### 11.1 评测维度（Benchmark-300）

| 维度 | 目标值 |
|---|---|
| 语义解析准确率（scene / trip_rigidity 对齐率） | ≥ 95% |
| 逻辑冲突率（时间逆序/距离超限/推荐闭馆 POI 等） | = 0% |
| 软锁/硬锁破坏率 | = 0% |
| User Pin 破坏率（新增） | = 0% |
| Mock 数据作用域违规率（幻觉 POI） | = 0% |
| 模式权重差异正确率（轻/全托管评分差异是否符合规则） | ≥ 90% |

### 11.2 CI/CD 评测门禁

每次 Prompt 变更 → 自动运行 Benchmark-300 → 新版本所有指标 ≥ 旧版本 → 才允许灰度发布

### 11.3 BadCase 逆向捕获

白名单内测阶段：用户点击"撤销方案"或频繁手动替换 → 完整 Context 日志自动标记为 BadCase → AI PM 每周五清洗并转化为新 TestCase → 充实评测集闭环
